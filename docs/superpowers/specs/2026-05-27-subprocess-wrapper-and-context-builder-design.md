## 0. Why this sub-project exists

The orchestrator (sub-project 4) needs two things from this layer:

- A reliable way to **invoke a configured CLI tool** (claude / codex / gemini) for a given role (planner / designer / reviewer / verifier_c), with the timeout, retry, heartbeat, and JSON validation semantics the parent design pins under "CLI Invocation Reliability."
- A way to **build a per-role CONTEXT.md** that gives each role exactly the slice of claim-graph state it should see, per the redesign spec §6.2.

Sub-project 1 built the workspace layout and hooks. Sub-project 2 built the verifiers. Sub-project 3 builds the bridge between the on-disk state and the agent CLIs. Without this, the round loop in sub-project 4 has nothing to spawn.

## 1. Scope

In scope:

- New module `harness/spawn.py` with `RoleOutput` dataclass + `spawn_role` public function + internal heartbeat/dispatch helpers.
- New module `harness/context.py` with four `build_<role>_context` functions.
- New test file `tests/test_spawn.py` with ~25 tests covering happy path, retry, timeout, dispatch.
- New test file `tests/test_context.py` with ~15 tests covering per-role context content.
- New test fixture `tests/fixtures/fake_cli.py` — a small Python script that simulates the claude/codex/gemini CLIs with scripted scenarios (ok / nonzero / nonjson / slow / hang / validate_fail).

Out of scope (deferred to later sub-projects):

- The orchestrator round loop itself — sub-project 4 wires spawn_role and the context builders into the planner → designer → reviewer → verifier_c sequence.
- Writing the `rounds/<n>/scratch/<role>.json` output files — `spawn_role` returns a `RoleOutput` dataclass; the orchestrator decides where to persist parsed outputs.
- Cost / token accounting — parent design defers to post-v0 (no automatic cost ceiling in v0).
- Session resumption — parent design line 192 pins each spawn as fresh (no `--resume`). v0.1 may revisit.
- `repo/` and `sources/` corpora — parent design defers to v0.2+.

## 2. Public API surface

### 2.1 `harness/spawn.py`

```python
from dataclasses import dataclass
from pathlib import Path
from typing import Callable


@dataclass(frozen=True)
class RoleOutput:
    verdict: str                    # "ok" | "spawn-failed" | "timeout" | "output-parse-fail"
    parsed: dict | None = None      # parsed JSON dict on verdict=="ok"
    stderr_tail: str = ""           # last ~100 lines of stderr (for actions.jsonl)
    elapsed_seconds: float = 0.0
    retry_count: int = 0            # 0 or 1; matches the retry-once-on-* contract


def spawn_role(
    role: str,
    harness_config: dict,
    context_md: str,
    prompt: str,
    workspace_root: Path,
    round_id: str,
    variant_id: str | None,
    validator: Callable[[dict], None] | None = None,
) -> RoleOutput:
    """Spawn the configured CLI tool for `role`, send context+prompt, parse
    JSON output, optionally validate via `validator`. See §3 for full
    retry/timeout/heartbeat semantics."""
```

Internal helpers (module-private):
- `_run_with_heartbeat(cmd, stdin_text, spawn_timeout_seconds) -> _RunResult`
- `_invoke_claude(model, prompt_with_context) -> list[str]` (returns the Popen argv)
- `_invoke_codex(model, prompt_with_context) -> list[str]`
- `_invoke_gemini(model, prompt_with_context) -> list[str]`
- `_TOOL_INVOKERS: dict[str, Callable]` — dispatch table

## 3. Behavioral semantics

### 3.1 `spawn_role` execution flow

The function reads `harness_config["models"][role]` for the `tool` and `model` fields. If the role is missing from config, raises `KeyError`. If the tool is not in `_TOOL_INVOKERS`, raises `ValueError`.

The full flow:

1. **Setup**: Build `cmd = _TOOL_INVOKERS[tool](model, "")` (empty prompt, just the argv shape). Compose `stdin_text = context_md + "\n\n" + prompt`. Read `spawn_timeout_seconds` from `harness_config["run"]["spawn_timeout_seconds"]` (default 300).

2. **Write CONTEXT.md to scratch**: `workspace_root / "rounds" / round_id / "scratch" / f"{role}.context.md"`. This is for audit only; the actual delivery to the CLI is via stdin. The directory is gitignored.

3. **Pass 1 — invoke the CLI**:
   - Call `result1 = _run_with_heartbeat(cmd, stdin_text, spawn_timeout_seconds)`.
   - If `result1.verdict == "timeout"`: return `RoleOutput(verdict="timeout", ...)`. **Timeouts are not retryable** — a stuck or runaway CLI is a different failure mode than a vendor flake.
   - If `result1.returncode != 0`: **retry once**. `time.sleep(30)` (configurable for testing via a module-level constant), then `result2 = _run_with_heartbeat(cmd, stdin_text, spawn_timeout_seconds)`. If `result2.verdict == "timeout"` → `RoleOutput(verdict="timeout", retry_count=1, ...)`. If `result2.returncode != 0` → `RoleOutput(verdict="spawn-failed", retry_count=1, ...)`. Otherwise, `parse_target = result2.stdout`.
   - If `result1.returncode == 0`: `parse_target = result1.stdout`.

4. **Pass 2 — JSON parse + validate**:
   - `parsed = json.loads(parse_target)`. If this raises `json.JSONDecodeError`, OR if `validator(parsed)` raises (caller's choice of exception class), **retry once**:
     - Append to the prompt: `"\n\nYour previous output failed schema validation: <error message>. Output ONLY valid JSON matching the schema. No prose."`
     - Re-run the heartbeat call with the appended prompt.
     - If the retry's exit code is non-zero OR JSON-parses/validates fail again → `RoleOutput(verdict="output-parse-fail", retry_count=1, ...)`.
   - On success → `RoleOutput(verdict="ok", parsed=parsed, ...)`.

**Mutual exclusivity:** A non-zero exit prevents JSON parsing from even running. The retry_count field is 0 or 1, never 2 — there's at most one retry across the whole spawn.

### 3.2 `_run_with_heartbeat`

Single-call execution with heartbeat + spawn timeout. Returns a `_RunResult` with `(returncode, stdout: bytes, stderr_tail: str, elapsed: float, verdict: "ok"|"timeout")`.

Algorithm:

```
spawn_start = time.monotonic()
last_output = spawn_start              # threading.Lock-guarded
stdout_buf = bytearray()
stderr_lines = collections.deque(maxlen=100)

try:
    proc = subprocess.Popen(cmd, stdin=PIPE, stdout=PIPE, stderr=PIPE,
                            bufsize=0)
except FileNotFoundError:
    return _RunResult(returncode=127, stdout=b"",
                      stderr_tail=f"tool not on PATH: {cmd[0]}",
                      elapsed=0.0, verdict="ok")
                      # returncode=127 mirrors shell convention; spawn_role
                      # interprets non-zero exits uniformly

threading.Thread(target=_write_stdin, args=(proc, stdin_text), daemon=True).start()
threading.Thread(target=_read_stdout, args=(proc, stdout_buf, last_output_ref), daemon=True).start()
threading.Thread(target=_read_stderr, args=(proc, stderr_lines, last_output_ref), daemon=True).start()

while True:
    if proc.poll() is not None:
        break
    now = time.monotonic()
    if now - spawn_start > spawn_timeout_seconds:
        proc.kill()
        proc.wait()
        return _RunResult(verdict="timeout", ...)
    with lock:
        silence = now - last_output
    if silence > 90:
        proc.terminate()
        try:
            proc.wait(timeout=5)
        except subprocess.TimeoutExpired:
            proc.kill()
            proc.wait()
        return _RunResult(verdict="timeout", ...)
    time.sleep(1)

# Drain any remaining output
for t in reader_threads:
    t.join(timeout=1)

return _RunResult(returncode=proc.returncode, stdout=bytes(stdout_buf),
                  stderr_tail="\n".join(stderr_lines),
                  elapsed=time.monotonic() - spawn_start,
                  verdict="ok")
```

**The 1-second polling interval** balances responsiveness vs CPU. **The 100-line stderr cap** prevents pathological output from filling memory. **The 90-second silence threshold** is the parent design's pinned value.

**Threading note:** Daemon threads exit when the main thread exits. The 1-second join timeout on each reader is defensive — if a thread is genuinely stuck reading from a closed pipe, joining-with-timeout prevents the test suite from hanging.

### 3.3 Tool dispatch

`_TOOL_INVOKERS` is a module-level dict mapping tool name to an invoker function:

```python
_TOOL_INVOKERS = {
    "claude": _invoke_claude,
    "codex":  _invoke_codex,
    "gemini": _invoke_gemini,
}


def _invoke_claude(model: str) -> list[str]:
    return ["claude", "-p", "--output-format", "json", "--model", model]

def _invoke_codex(model: str) -> list[str]:
    return ["codex", "exec", "--model", model, "--json"]

def _invoke_gemini(model: str) -> list[str]:
    return ["gemini", "--model", model, "--output", "json"]
```

The exact flags above match the most common v0 CLI shapes; the prompt is fed via stdin in all three cases (parent design line 192 pins each spawn as fresh — no resume, no session ID). If a tool's actual CLI shape diverges, only the invoker function changes — the rest of `spawn.py` is tool-agnostic.

A v0 simplification: all three tools' invoker functions are pure (model → argv). Test mocking only needs to monkey-patch `_TOOL_INVOKERS["claude"]` (or others) to point at the fake CLI script.

### 3.4 Per-role CONTEXT.md content

All four CONTEXT.md outputs share a common header:

```markdown
# CONTEXT — <role>

Round: <round_id>
Variant: <variant_id>
Goal version: <goal_version from goal.toml>
Generated: <ISO 8601 timestamp>
```

After the header, role-specific sections per the redesign spec §6.2:

**Planner (lightest):**

```markdown
## Registered decisions

| ID | Question | Status |
|---|---|---|
| ... |

(from derived/decisions.json, filtered to status in {open, proposed})

## Stale proposals

(present only if detect_stale_proposals returns non-empty; otherwise omitted)

## Recent rejections (this variant)

(last 3 rj-*.md files filtered by variant; one-line summary each)
```

**Designer:**

```markdown
## Registered decisions

(same shape as planner)

## Positions you have committed to in <variant_id>

| Decision | Position | Claim ID |
|---|---|---|
| ... |

(from variant's claims/cl-*.json files; only decision-type claims with a position)

## Your pending proposals

| Proposed decision ID | Question | Rationale |
|---|---|---|
| ... |

(from this variant's claims/cl-*.json `proposed_decision` payloads where the
 id is NOT yet in derived/decisions.json)

## Slug discipline

(verbatim copy of constitution.md's "Slug discipline" section)
```

**Reviewer (heaviest):**

```markdown
## Registered decisions

(same as planner)

## All positions in use across variants

| Decision | Variant | Position | Claim ID |
|---|---|---|---|
| ... |

(grouped by decision_id; one row per (variant, decision_id, position) tuple
 across all variants under variants/nodes/v-*)

## Pending designer proposals (this round)

(from this round's designer.json output; empty if designer emitted no
 proposed_decision payloads this round)

## Recent canonicalizations (last 5 rounds)

(output of `git log --grep="^Action: canonicalize" -n 5 --format="%h %s"`)

## Registry posture

registry_size: <N>
bootstrap_threshold: <T>
posture: <"bootstrap-permissive" if N < T else "default-deny">

(reads [claim_graph].bootstrap_registry_size_threshold from harness.toml,
 default 5; gives the reviewer the rule from the constitution explicitly)
```

**Verifier C (lightest):**

```markdown
## Registered decisions

(same as planner — for context only; Verifier C reads the doc patch and
 cited evidence files, not the broader claim graph)
```

### 3.5 Failure aggregation, audit, and side-effects

`spawn_role` writes `rounds/<round_id>/scratch/<role>.context.md` regardless of verdict, so a post-mortem can read what the agent actually saw. The scratch directory is gitignored per the workspace template; never committed.

`spawn_role` does NOT write `rounds/<n>/scratch/<role>.json` — that's the orchestrator's concern (sub-project 4) once it has the `RoleOutput` in hand. Reason: keeping spawn_role purely about "invoke and parse" makes it easier to unit-test.

`spawn_role` does NOT log to `actions.jsonl`. Same reason — the orchestrator owns the append-only diagnostic log.

## 4. Test plan

### 4.1 Test fixture: `tests/fixtures/fake_cli.py`

A standalone executable Python script (~60 LOC) with `--scenario` flag:

| Scenario | Behavior |
|---|---|
| `ok` | Read stdin, write `{"ok": true, "echo": "<sha256-of-stdin>"}` to stdout, exit 0 |
| `nonzero` | Write `"boom"` to stderr, exit 1 |
| `nonjson` | Write `"not json output"` to stdout, exit 0 |
| `slow N` | Write a heartbeat line to stderr every 100ms for N seconds, then `{"ok": true}` + exit 0 |
| `hang` | Read stdin, then sleep forever (no output) |
| `validate_fail` | Write `{"wrong_field": "x"}` to stdout, exit 0 |
| `transient` | First invocation: exit 1. Second invocation: exit 0 with `{"ok": true}`. Uses a per-process-tree marker file in TMPDIR. |

Tests invoke it via `_TOOL_INVOKERS["claude"] = lambda model: ["python3", "tests/fixtures/fake_cli.py", "--scenario", "ok"]` (or similar), then call `spawn_role` and assert.

## 5. Edge cases

- **Concurrent spawns:** v0 is single-threaded sequential — only one `spawn_role` call active at a time per process. No locking concern for the scratch CONTEXT.md path because each (round_id, role) combination is unique.
- **`harness_config` is mutated by caller:** `spawn_role` reads it but does not mutate it. Callers can pass dataclass-converted configs or raw `tomllib.loads` dicts interchangeably.
- **`validator` raises an unexpected exception type:** caught and treated as a parse failure. The exception's `str()` is included in the retry prompt.
- **A round directory doesn't exist yet:** `spawn_role` creates `rounds/<round_id>/scratch/` if missing (parent dirs included). Caller doesn't need to pre-create.
- **A reviewer's `registered decisions` is empty:** the rendered table is omitted with a "No registered decisions yet." note. Bootstrap-permissive posture is the natural fallback.
- **The fake CLI's `transient` scenario** uses a per-process-tree TMPDIR marker because pytest may invoke the same test multiple times — the marker is reset per test by `tearDown`.
- **A CONTEXT.md exceeds Claude's context window:** out of scope for v0 (Claude Opus 4.7 has a 1M context window in this project). v0.1 may add truncation.

## 8. Success criteria

This sub-project ships when all of these are true:

1. `from harness.spawn import spawn_role, RoleOutput` and `from harness.context import build_planner_context, build_designer_context, build_reviewer_context, build_verifier_c_context` work.
2. `spawn_role` correctly dispatches by `harness_config["models"][role]["tool"]` to one of the three CLI shapes.
3. The 90-second-stderr-silence heartbeat triggers SIGTERM then SIGKILL on a truly stuck process — verified by a test with a shorter custom threshold.
4. JSON parse failure and validator failure both retry once with an appended hint prompt.
5. Non-zero exit retries once after a sleep; both failures return `verdict="spawn-failed"` with `retry_count=1`.
6. `verdict="timeout"` is not retryable.
7. Each role's CONTEXT.md contains exactly the sections from redesign spec §6.2 — no more, no less.
8. All ~258 tests pass under `python3 -m unittest discover tests/`.

## 10. What's NOT in v0 (deferred)

- **Session resumption** — parent design pins fresh-spawn-per-role. v0.1 may revisit with --resume.
