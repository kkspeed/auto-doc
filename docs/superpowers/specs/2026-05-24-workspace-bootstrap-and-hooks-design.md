## 0. Why this sub-project exists

The orchestrator's first runnable artifact is a scaffolded workspace. Before anything in the round loop can fire, a fresh project directory needs:

- A populated `workspace_template/` layout copied in (currently only `constitution.md` + `goal.toml` exist — `harness.toml`, `seed_doc.md`, `.gitignore`, and the `hooks/` directory are still missing).
- A configured git repository (`git init` + `git config core.hooksPath workspace/hooks/`).
- Active hooks that catch malformed agent output BEFORE it lands in git history — both file-local validation (pre-commit) and commit-shape validation (commit-msg) per the parent design's closed-vocab discipline.

This sub-project ships those four artifacts: `harness/cli.py` (init subcommand), the missing workspace template files, the pre-commit hook, and the commit-msg hook. Together they produce a runnable workspace skeleton that the round loop (sub-project 4) and verifiers (sub-project 2) will plug into.

## 1. Scope

In scope:

- `harness/cli.py` — argparse-based CLI module with `init` subcommand; `python -m harness init <dir> [--reactivate]`.
- `workspace_template/harness.toml` — heavily commented orchestrator config (model + tool pins, run-loop bounds, claim-graph thresholds).
- `workspace_template/seed_doc.md` — empty stub with explanatory header.
- `workspace_template/.gitignore` — `CONTEXT.md`, `derived/`, `rounds/*/scratch/`, `*.tmp`, `repo/`, `sources/*/cache/`.
- `workspace_template/hooks/pre-commit` — executable Python script: cl-*.json schema validation, at-*.json schema validation, cross-field validators (decision_id resolution, vacuous slug, alias rewriting), citation resolution.
- `workspace_template/hooks/commit-msg` — executable Python script: closed-vocab trailer validation, required-trailers-per-Action enforcement, Action-aware file-set whitelist, scope check (≤ N sections per merge commit), decided-section immutability check.
- Tests: parity test for pre-commit validators, fixture-driven tests for both hooks, fixture-driven tests for `harness init`.

Out of scope (deferred to later sub-projects):

- `harness run` and `harness resume` subcommands (sub-projects 4 and 6).
- Verifier A and Verifier B implementations (sub-project 2).
- Subprocess wrapper for claude/codex/gemini CLIs (sub-project 3).
- PreToolUse hook (deferred to v0.1 per parent design).
- `harness status`, `harness brief` subcommands (deferred to v0.1).
- Evidence frontmatter schema validation (the pre-commit hook reads `superseded_by` field but does not enforce the full evidence frontmatter schema; that lands when sub-project 3's evidence-ledger writer is built).

## 3. Behavioral semantics

### 3.1 `harness init` CLI

```
harness init <dir> [--reactivate]
```

**Default behavior:**

1. Validate target: `<dir>` must not exist, OR must exist and be empty. Anything else → exit 1 with `harness init: refusing to clobber non-empty <dir>`.
2. Create `<dir>` if needed.
3. Copy the entire contents of `workspace_template/` into `<dir>`, preserving directory structure and executable bits on `hooks/*`.
4. Run `git init <dir>` (creates `<dir>/.git/`).
5. Run `git -C <dir> config core.hooksPath hooks/`. (v0 simplification: hooks live at `<dir>/hooks/` rather than the parent design's nested `<dir>/workspace/hooks/`. The parent design's project-root/workspace/ split was driven by anticipated `repo/`, `sources/`, and `.harness/` siblings that v0 doesn't have; flattening the layout removes a level of indirection without losing forward-compatibility — if those siblings land in v0.2+, they can move under a `workspace/` subdirectory then.)
6. Stage the initial scaffold and commit with message:
   ```
   harness: scaffold workspace

   Action: init
   ```
7. Print success: `harness init: workspace ready at <dir>`. Exit 0.

**`--reactivate` behavior:**

For users who cloned a workspace from elsewhere (where `.git/` exists but `core.hooksPath` is not set in the clone). Skips steps 1-4 and 6; only re-runs step 5. Exits with `harness init --reactivate: hooks reactivated at <dir>` on success, or `<dir> is not a git repository` on failure.

### 3.5 Pre-commit hook

**File:** `workspace_template/hooks/pre-commit`, shebang `#!/usr/bin/env python3`, executable.

**Design constraint:** Self-contained — does NOT `import harness.claim_graph`. Duplicates `SchemaError`, `RegistryInvariantError`, the closed enums (`CLAIM_TYPES`, `AT_TYPES`, `CANONICALIZATION_KINDS`, `CONFIDENCES`, `DECISION_STATUSES`, `VACUOUS_POSITION_SLUGS`, `SLUG_REGEX`), validation helpers (`_require`, `_require_enum`, `_require_slug`), the `Claim` and `Attack` dataclasses' `from_dict` paths, and the three cross-field validators (`validate_claim_decision_id_resolution`, `validate_claim_position_not_vacuous`, `validate_claim_position_not_alias`) verbatim. A parity test enforces drift safety (§4).

**Reason for not importing:** parent design's inline-duplication-over-import-coupling decision — importing `harness.*` would couple hook execution to whatever venv was active when `git commit` fires, and to the filesystem layout of the harness install. The hook must work under any Python 3.11+ on PATH with no extra setup.

**Checks (all errors collected; reported together; nonzero exit if any error):**

1. **cl-*.json schema validation** — for every staged `variants/nodes/v-*/claims/cl-*.json` (added or modified), load JSON, call duplicated `Claim.from_dict(...)`. Propagated `SchemaError` becomes a hook error.
2. **at-*.json schema validation** — same shape, runs duplicated `Attack.from_dict(...)`.
3. **Cross-field validators** — for each staged cl-*.json: run all three duplicated validators against `derived/decisions.json` (loaded once at hook start; falls back to empty registry if missing) and `derived/canonical_slug_registry.json` (also loaded once; empty if missing).
4. **Citation resolution** — for each staged `.md` file, extract additions from `git diff --cached -U0 -- <file>`; parse `[^ev-NNNNNN]` tokens; for each unique cite:
   - Verify `evidence/ev-NNNNNN.md` exists; if not → hook error.
   - Read the file's TOML frontmatter; if `superseded_by` key is present and not empty → hook error.

**What pre-commit does NOT do** (handled by commit-msg in §3.6):
- Trailer schema validation.
- Scope check.
- Decided-section immutability check.

**Exit codes:**
- 0 — all checks passed.
- 1 — at least one check failed. stderr contains a bullet list of failures with file paths and one-line explanations.

### 3.6 commit-msg hook

**File:** `workspace_template/hooks/commit-msg`, shebang `#!/usr/bin/env python3`, executable.

Git invokes commit-msg with `$1` = path to the commit message file. The hook reads that file, parses trailers, then performs Action-aware checks.

**Trailer parsing:** Standard Git trailer format (`Key: Value` lines in a trailing block separated from the body by a blank line). Parsed via hand-rolled regex (no `git interpret-trailers` subprocess dependency); standard trailers `Signed-off-by` and `Co-authored-by` pass through untouched.

**Closed vocab enforced:**

| Trailer key | Allowed values |
|---|---|
| `Action` | `init`, `merge`, `register-decision`, `canonicalize`, `registry-sync`, `reviewer-rejected`, `phase-a-fail`, `phase-b-fail`, `phase-c-dispute`, `spawn-failed`, `output-parse-fail` |
| `Reason` | `uncited-claim`, `cross-field-fail`, `vacuous-position`, `proposal-rejected`, `scope-violation`, `immutability-violation`, `phantom-claim`, `dangling-evidence`, `silent-goal-toml-edit` |
| `Variant` | regex `^v-\d{3}$` |
| `Round` | regex `^round-\d{6}$` |
| `Reviewer` | regex `^v-\d{3}$` |

Any unknown trailer key OR value outside its enum → reject with a specific error.

**Required-trailers matrix:**

| Action | Variant | Round | Reason | Reviewer |
|---|---|---|---|---|
| `init` | – | – | – | – |
| `merge` | required | required | – | – |
| `register-decision` | – | – | – | – |
| `canonicalize` | – | – | – | – |
| `registry-sync` | – | – | – | – |
| `reviewer-rejected` | required | required | required | required |
| `phase-a-fail` / `phase-b-fail` / `phase-c-dispute` | required | required | required | – |
| `spawn-failed` / `output-parse-fail` | required | required | – | – |

**Action-aware file-set whitelist:**

The hook reads the staged file list (`git diff --cached --name-only`) and ensures every file matches one of the Action's allowed paths:

- `init` — no restriction.
- `merge` — `variants/nodes/v-*/doc/*.md`, `variants/nodes/v-*/claims/cl-*.json`, `variants/nodes/v-*/attacks/at-*.json`, `evidence/ev-*.md`, `rejections/rj-*.md`, `variants/nodes/v-*/scorecard.json`, `actions.jsonl`.
- `register-decision` — `goal.toml`, `derived/decisions.json`, `actions.jsonl`.
- `canonicalize` — `variants/nodes/v-*/claims/cl-*.json`, `derived/canonical_slug_registry.json`, `actions.jsonl`.
- `registry-sync` — `variants/nodes/v-*/doc/*.md`, `derived/decisions.json`, `derived/canonical_slug_registry.json`, `actions.jsonl`.
- Failure actions (`reviewer-rejected`, `phase-*-fail`, `spawn-failed`, `output-parse-fail`) — `rejections/rj-*.md`, `actions.jsonl`.

Any staged file outside its Action's whitelist → reject with `scope-violation`.

**Scope check (Action: merge only):**

Count distinct files matching `variants/nodes/v-*/doc/*.md` in the staged set. Read `[run].patch_max_sections` from `harness.toml` (default 3). If count > limit → reject with `scope-violation`. Error message includes count, limit, and the offending file paths.

**Decided-section immutability check (applied to every staged doc section):**

For each staged `variants/nodes/v-*/doc/*.md`:

1. Get HEAD content: `git show HEAD:<path>`. If the file is new (no HEAD version) → no immutability constraint, skip.
2. Parse frontmatter `tags` array from HEAD content (regex on `tags = [...]`). If `"decided"` is NOT in tags → section was not yet decided → skip.
3. Get staged content: `git show :<path>` (or read the working tree; both equivalent at commit-msg time).
4. Compare:
   - If only the `tags` array differs (frontmatter rest + body unchanged) → "tag-only" change. Allowed iff `Action: registry-sync`. Otherwise reject with `immutability-violation`.
   - If anything beyond `tags` differs → "body change". Allowed iff `goal.toml` is also in the staged set AND its `goal_version` field differs from HEAD (concurrent goal-version bump). Otherwise reject with `immutability-violation`.

**Exit codes:**
- 0 — all checks passed.
- 1 — at least one check failed.

### 4.2 `tests/test_hook_parity.py` — pre-commit ↔ claim_graph.py parity

Loads each fixture (described below) and runs BOTH the inlined pre-commit validators AND the canonical `harness.claim_graph` validators. Asserts identical pass/fail verdict and identical exception type + message substring.

Test methods (one per schema family):
- `test_claim_validator_parity` — exercises every Claim-from-dict path on ~10 fixtures (valid + invalid).
- `test_attack_validator_parity` — same for Attack on ~6 fixtures.
- `test_cross_field_validator_parity` — exercises decision_id resolution, vacuous, alias on ~8 fixtures.

Mechanism: the test imports the canonical validators from `harness.claim_graph`, and imports the inlined copy by exec-loading the pre-commit script in a sandboxed module namespace (the script's `if __name__ == "__main__":` guard prevents auto-run). Both run on the same dict inputs; verdicts compared.

~3 test methods covering ~24 fixtures.

### 4.3 `tests/test_pre_commit_hook.py` — integration tests against a temp git repo

Each test sets up a minimal workspace under tempdir, stages a known file set, runs the pre-commit script as a subprocess (so PATH-only Python execution is exercised), and asserts exit code + stderr contents.

- `test_passes_on_valid_claim_and_attack` — staged cl + at + md file, all valid → exit 0.
- `test_rejects_invalid_claim_json` — staged cl with bad enum → exit 1; stderr names the file + the field.
- `test_rejects_invalid_attack_json` — staged at with missing required field → exit 1.
- `test_rejects_vacuous_position_slug` — cl with position=`"tbd"` → exit 1.
- `test_rejects_alias_slug_as_position` — cl with position equal to an alias key in registry → exit 1.
- `test_rejects_decision_id_for_retired_decision` — cl pointing at retired decision (without proposed_decision) → exit 1.
- `test_passes_cite_resolution_for_existing_evidence` — md with `[^ev-000001]` and evidence/ev-000001.md exists, no superseded_by → exit 0.

## 5. Edge cases

- **`harness init` with a target dir containing only `.DS_Store`**: not strictly empty; reject per the "must be empty" rule. Documented in error message.
- **Pre-commit hook on a workspace without `derived/`**: cross-field validators degrade gracefully — treat missing `decisions.json` and `canonical_slug_registry.json` as empty registries. Validators still fire; some may be no-ops (e.g., alias rewriting check skips when registry is empty per existing `validate_claim_position_not_alias` semantics).
- **commit-msg hook on a commit with NO trailers (human runs `git commit` manually with just a subject line)**: missing Action → reject with clear instructions: "every commit must carry an Action trailer; see workspace/constitution.md".
- **Hook receives a commit that modifies a section file whose HEAD content has malformed frontmatter**: the regex parse for `tags` returns empty → treated as "not decided" → immutability check passes. Hook does not validate frontmatter-completeness on HEAD content (that's the orchestrator's responsibility on write).
- **`harness init` invoked twice on the same dir**: second invocation fails because the dir is no longer empty (it has the scaffold). User can `--reactivate` instead, or `rm -rf <dir>` and re-init.
- **commit-msg whitelist for `Action: merge` includes `actions.jsonl`** — every Action's whitelist does, since the orchestrator appends to actions.jsonl on every commit.
- **Citation resolution on a brand-new markdown file** (no HEAD version): `git diff --cached -U0` returns the entire content as additions → all cites in the file get checked. Correct behavior.

## 7. Migration / compatibility

- No changes to existing `harness/claim_graph.py` — pre-commit hook duplicates validators inline; parity test enforces drift safety.
- No changes to existing workspace_template files (`constitution.md`, `goal.toml`) — only additions.
- Pre-commit's reads of `derived/decisions.json` and `derived/canonical_slug_registry.json` use the existing on-disk JSON shape produced by `harness/claim_graph.py:dump_decisions_to_json` (and the registry's `to_dict()`). If those formats evolve, the hook's parser updates with them — covered by parity tests at the dataclass-loading level.
- harness.toml format may evolve in v0.1 (e.g., add per-tool authentication fields); the v0 template is forward-compatible because TOML ignores unknown sections by default.

## 8. Success criteria

This sub-project ships when all of these are true:

1. `python3 -m harness init /tmp/test-workspace` produces a fully-scaffolded workspace at `/tmp/test-workspace` with the template files, an initialized git repo, configured hooksPath, and an initial commit carrying the `Action: init` trailer. (Covers parent design SC1.)
2. The same command refuses to clobber a non-empty existing directory.
3. The pre-commit hook, invoked on a workspace containing a malformed cl-*.json (e.g., position slug = `"tbd"`), rejects the commit with `Reason: vacuous-position` in stderr.
4. The pre-commit hook, invoked with a markdown file citing a non-existent evidence id, rejects the commit. (Partially covers parent design SC4 — full SC4 requires the orchestrator's evidence-ledger writer in sub-project 3.)
5. The commit-msg hook rejects a commit attempting to modify a decided section's body without a concurrent `goal_version` bump.
6. The commit-msg hook rejects a commit whose `Action: merge` trailer is paired with a staged set touching 4+ doc section files.
7. All ~162 tests pass under `python3 -m unittest discover tests/`.
8. The parity test (`test_hook_parity.py`) passes — inlined validators match canonical validators on all fixtures.

## 10. What's NOT in this sub-project (deferred)

- **`harness run` and `harness resume` subcommands** — sub-projects 4 and 6 add them to `harness/cli.py`. The `cli.py` module is structured (argparse subparsers) so new subcommands plug in cleanly.
- **PreToolUse hook** — parent design defers to v0.1 alongside the full role-isolation matrix.
- **PATH detection of the CLI tools named in `harness.toml`** — `harness init` writes defaults without checking they're installed. Sub-project 4 (round loop) is the natural home for "fail fast if `codex` not on PATH but configured as verifier_c's tool" — at `harness run` startup.
- **Evidence frontmatter schema validation** — pre-commit's cite resolution only reads the `superseded_by` field, not the full evidence schema. Full validation lands when sub-project 3 builds the evidence-ledger writer (which owns the schema authoritatively).
- **`harness status`, `harness brief`** — parent design defers to v0.1.
- **Hook performance optimization** — at very large workspaces (hundreds of staged files), the hook walks every file and runs `git diff --cached -U0` per markdown file. Acceptable for v0 where rounds touch ≤ 3 sections. Sub-project 5+ can revisit if real overnight runs surface latency.
- **PyPI release + harness install via `pip install`** — parent design defers; the v0 install path is `git clone` + `python -m harness ...`.
