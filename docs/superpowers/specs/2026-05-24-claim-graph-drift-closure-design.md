## 0. Why this closure exists

A drift audit of `harness/claim_graph.py` against the redesign spec found three gaps:

1. **Two §5.3 morning_brief sections have no render helper.** The spec calls for five sections; the module ships three (`position collisions`, `decisional asymmetry`, `pending registry changes`). Missing: `canonicalizations applied this round` and `stale proposals`. The matching plan's spec-coverage table only listed the three implemented sections — the drift originated in the plan and propagated into the code.
2. **Flow D (decision_id canonicalization) has no apply-side function.** Spec §4.5 says: after the human edits `goal.toml` for a decision_id rename + bumps `goal_version`, "orchestrator runs registry-sync + rewrites cl-*.json/at-*.json fields." `render_pending_registry_changes` renders the queue (so the human ritual works), but no function performs the cascade. The same mechanical class of work that `apply_canonicalization` owns for positions belongs in this module for decision_ids.
3. **`register_decision` does not regenerate `derived/decisions.json`.** Spec §4.1 step 6 describes the `register-decision` action as three things: append to goal.toml, bump goal_version, regenerate decisions.json. The function does the first two. If the orchestrator forgets the third, goal.toml advances while decisions.json stays stale.

This closure is additive and small: ~150 LOC across one source file and three test files, no new modules, no schema changes.

## 3. Behavioral semantics

### 3.1 `apply_decision_id_canonicalization`

Cascades a decision_id rename across the whole ledger. The caller has already observed the human's goal.toml edit + goal_version bump, has reloaded `decisions` from the new goal.toml, and resolved the `(from_id, to_id)` pair from a queued Flow D canonicalization proposal.

**Execution order (rails-first, identical pattern to `apply_canonicalization`):**

1. **Pre-flight rails** (all raise `SchemaError` on violation; no filesystem mutation):
   - `from_id != to_id`
   - Both `from_id` and `to_id` pass `_require_slug`
   - `to_id in decisions` (rename target must be registered by the human's edit)
   - `from_id not in decisions` (presence means the caller jumped the gun)
2. **Filesystem walk** (collect rewrite records, write each file at most once):
   1. `variants/nodes/v-*/claims/cl-*.json`
      - `decision_id == from_id → to_id`
      - `section_id == from_id → to_id` (section_id often shadows decision_id in v0)
      - `proposed_decision.id == from_id → to_id` when present
   2. `variants/nodes/v-*/attacks/at-*.json`
      - `target_decision_id == from_id → to_id` (for `propose_decision_cut`)
      - `scope == from_id → to_id` (for `propose_canonicalization` with `kind=position`)
   3. `variants/nodes/v-*/doc/*.md`
      - Frontmatter `section_id == from_id → to_id`, using the same TOML-frontmatter handling as the existing retag walker
      - Section tags (`decided`, `unresolved`) are NOT changed — Flow D is a rename, not a retirement
3. **Registry mutation** (last, after files succeed):
   - If `registry.data[from_id]` exists, move it to `registry.data[to_id]`
   - If `registry.data[to_id]` already exists with non-empty `canonical` or `aliases`, raise `RegistryInvariantError("cannot merge two non-empty registry entries")`
   - If `registry.data[to_id]` exists but is empty (e.g. orchestrator pre-created it), the move overwrites the empty entry
   - If `registry.data[from_id]` does not exist, the registry move is a no-op (some decisions never had positions yet)

**Return shape:**

```python
{
  "claims_rewritten":   [{"path", "claim_id", "field", "from", "to"}, ...],
  "attacks_rewritten":  [{"path", "attack_id", "field", "from", "to"}, ...],
  "sections_rewritten": [{"path", "section_id", "from", "to"}, ...],
  "registry_moved":     bool,
}
```

Each rewrite record carries the `field` that was changed so callers (commit trailer authoring, morning_brief render) can report precisely.

**Idempotent no-op:** If `from_id` appears nowhere (all walks return empty lists AND registry has no `from_id` key), returns the report with empty lists and `registry_moved: False`. Does not raise. Allows safe retry after partial failure mid-walk.

### 3.2 `render_canonicalizations_applied`

Renders the `## Canonicalizations applied this round` section of `morning_brief.md`. Accepts two lists:

- `position_rewrites` — same shape as the records produced by `apply_canonicalization` (`{"path", "claim_id", "decision_id", "from", "to"}`)
- `decision_id_rewrites` — synthesized by the orchestrator from `apply_decision_id_canonicalization`'s return value, with shape `{"from", "to", "kind", "paths"}` where `kind ∈ {"claim", "attack", "section"}` and `paths` is the list of touched files

Returns `"## Canonicalizations applied this round\n\nNo canonicalizations applied this run.\n"` when both lists are empty. Otherwise renders two sub-tables (one per kind) under the level-2 header, matching the markdown shape of the existing renderers.

### 3.3 `render_stale_proposals_table`

Renders the `## Stale proposals` section. Input shape comes directly from `detect_stale_proposals`: `[{"decision_id", "question", "rounds_since_proposal", "introduced_round"}, ...]`. Returns the friendly empty line on empty input. Rows sorted by `decision_id` (matches the detector's existing output ordering).

### 3.4 `register_decision` atomicity

New optional parameter `decisions_json_path`. When provided:

1. Read goal.toml + load existing decisions (unchanged)
2. Build appended blocks + bumped version (unchanged)
3. Write goal.toml (unchanged)
4. **New:** Reload `(decisions, version)` from the freshly-written goal.toml and call `dump_decisions_to_json(decisions, version, decisions_json_path)`

Reloading rather than building the in-memory dict avoids drift between the TOML text just written and the dataclass shapes the caller might have constructed differently.

When `decisions_json_path` is `None` (existing callers, existing tests), behavior is unchanged.

## 5. Edge cases

- **`from_id` exists in registry but `to_id` doesn't:** registry move creates `registry.data[to_id]` with the moved contents (canonical list + aliases preserved verbatim).
- **`from_id` and `to_id` both registered in `decisions` (Caller error: human did NOT delete the old entry):** rail check `from_id not in decisions` catches this. Raise `SchemaError`.
- **Section frontmatter has multiple `section_id =` lines (malformed):** the existing `_section_decision_id` matcher catches only the first; we leave the rest. Consistent with retag walker's defensive read-only-first-match behavior.
- **`proposed_decision` block in cl-*.json carries a different id than `decision_id`:** that's a separate validation concern (already covered by `validate_claim_decision_id_resolution`); Flow D apply only rewrites when the inner id equals from_id, leaving other ids alone.
- **No variants directory exists yet (early bootstrap):** the walk returns empty lists; registry move proceeds if applicable; report shows empty `claims_rewritten` etc. Function does not raise.

## 8. Success criteria

This closure ships when all of these are true:

1. All ~126 tests pass under `python3 -m unittest discover tests/`.
2. `harness/claim_graph.py` exports the three new functions and the modified `register_decision`.
3. `apply_decision_id_canonicalization` enforces all four rails from §3.1 step 1.
4. `apply_decision_id_canonicalization` is idempotent on no-op (verified by `test_idempotent_when_from_id_absent_everywhere`).
5. `register_decision`'s `decisions_json_path` parameter, when omitted, reproduces existing behavior exactly (verified by `test_path_argument_omitted_preserves_existing_behavior`).
6. The two new renderers produce the friendly empty-state line on empty input (parallels the existing renderers' contract).

## 10. What's NOT in this closure (deferred)

- **Orchestrator call sites for Flow D apply:** The function exists in this closure; wiring it into the round state machine (which step runs it, how it's triggered by `goal_version` bump + queued canonicalization proposal) belongs to the orchestrator plan.
- **Commit trailer schema for `Action: registry-sync` (which the orchestrator will use for Flow D commits):** Trailer enums live in the orchestrator/hook layer, not in `claim_graph.py`.
- **Pre-commit hook duplication:** No new validators are introduced (the rails are runtime checks inside `apply_decision_id_canonicalization`, not commit-time invariants on file contents). The existing hook-duplication-of-validators concern is unchanged.
