## 0. Why this sub-project exists

The actual harness loop. This is what the parent design's "harness run" is — the multi-hour autonomous loop that ties everything together. It consumes everything sub-projects 1-3 produced:

- Sub-project 1: workspace layout, pre-commit + commit-msg hooks, `harness init` CLI.
- Sub-project 2: pure-Python verifiers (verify_citation_completeness, verify_cite_resolution, verify_excerpt_match).
- Sub-project 3: subprocess wrapper (spawn_role) + per-role CONTEXT.md builders.

And calls into the existing `harness/claim_graph.py` for:
- `register_decision` (Flow A — designer-proposed new decisions get auto-registered after reviewer approval).
- `apply_canonicalization` (Flow C — high-confidence position canonicalizations get auto-applied).
- `apply_reviewer_decision_proposals` (Flow A gating).
- `detect_position_collisions` / `detect_decisional_asymmetry` / `detect_stale_proposals` (post-round detectors).
- All the dataclass schemas (Claim, Attack, Decision, DecisionProposalVerdict, CanonicalSlugRegistry).

Sub-project 4 ships the round state machine that drives the per-round role sequence (planner → designer → Verifier A → Verifier B → reviewer → Verifier C → Flow A → Flow C → merge), the multi-round run loop with sequential variant rotation, the per-failure rj-*.md construction, the `actions.jsonl` append-only log, and the `harness run` CLI subcommand.

This sub-project closes Success Criterion 2 ("one overnight run completes without intervention") and contributes to SC4 (citation enforcement), SC5 (excerpt match), SC6 (variants disagree productively), and SC7 (crash recovery — partial; full resume is sub-project 6).

In scope:

- `harness/orchestrator.py` — RoundOutcome dataclass + run_round (linear per-round flow with early returns on rejection) + run_loop (multi-round with variant rotation and dual stopping caps).

Out of scope (deferred to later sub-projects):

- Morning brief pipeline + scorecard.json — sub-project 5.

## 2. Public API surface

### 2.1 `harness/orchestrator.py`

```python
from dataclasses import dataclass, field
from pathlib import Path


@dataclass(frozen=True)
class RoundOutcome:
    round_id: str                   # "round-000042"
    variant_id: str                 # "v-001"
    verdict: str                    # "merge" | "reviewer-rejected" | "phase-a-fail" |
                                    # "phase-b-fail" | "phase-c-dispute" |
                                    # "spawn-failed" | "output-parse-fail"
    reason: str | None = None       # closed-vocab Reason value when applicable
                                    # ("uncited-claim", "dangling-evidence",
                                    # "cross-field-fail", "proposal-rejected", etc.)
    rj_id: str | None = None        # "rj-000017" when a rejection was written
    elapsed_seconds: float = 0.0
    spawn_counts: dict = field(default_factory=dict)  # {role: 1 + retry_count}


def run_round(
    workspace_root: Path,
    harness_config: dict,
    round_id: str,
    variant_id: str,
) -> RoundOutcome:
    """Execute one round on one variant. Linear flow with early returns on any
    rejection path. See §3 for full phase semantics."""


def run_loop(
    workspace_root: Path,
    harness_config: dict,
    max_rounds: int | None = None,
    max_wall_clock_hours: float | None = None,
    variant_count: int = 2,
) -> list[RoundOutcome]:
    """Drive rounds in sequence with variant rotation. Stops at whichever cap
    fires first. At least one of max_rounds / max_wall_clock_hours required.
    Returns the full list of round outcomes for the caller to summarize."""
```

def write_role_scratch(
    workspace_root: Path,
    round_id: str,
    role: str,
    parsed: dict,
) -> Path:
    """Write rounds/<round_id>/scratch/<role>.json from a spawn_role's parsed

    variant_id: str,
    reason_class: str,
    failed_phase: str,
    detail: str,
    reviewer_id: str | None = None,

def append_actions_log(workspace_root: Path, entry: dict) -> None:

    Atomicity: each call writes `json.dumps(entry) + "\\n"` in a single
    `open(..., 'a')` + `write()` + `flush()` cycle. Crash mid-write may leave
    a partial line; sub-project 6's resume trims it before resuming."""

    workspace_root: Path,
    new_decision_ids: list[str],
) -> None:
    """git add goal.toml derived/decisions.json actions.jsonl && git commit

    'Action: register-decision'. Hook (sub-project 1) validates the file set."""


def commit_canonicalize(
    workspace_root: Path,
    rewrites: list[dict],
) -> None:
    """git add changed cl-*.json files + derived/canonical_slug_registry.json

def commit_merge(
    workspace_root: Path,
    round_id: str,
    variant_id: str,
    section_paths: list[str],
    claim_paths: list[str],
    attack_paths: list[str],
    evidence_paths: list[str],

    """git add all materialized files + actions.jsonl && git commit with

    round_id: str,
    variant_id: str,
    rj_id: str,
    reason: str,
    reviewer_id: str | None = None,
) -> None:
    """git add rejections/<rj_id>.md + actions.jsonl && git commit with the
    failure-class Action trailer (reviewer-rejected | phase-a-fail | etc.)

## 3. Behavioral semantics

### 3.1 Round phase sequence (`run_round`)

The flow is linear with early returns on any rejection path. Each phase, on failure:
1. Writes role scratch JSON (if a spawn happened).
2. Writes the rj-*.md.
3. Commits the rejection (failure-class Action trailer + Variant + Round + Reason).
4. Returns a RoundOutcome with the verdict.

**Setup (before phase 1):**
- `scratch_dir = workspace_root / "rounds" / round_id / "scratch"` — created (parents=True, exist_ok=True), gitignored.
- `spawn_counts: dict[str, int] = {}` — incremented per spawn.
- `start_ts = time.monotonic()`.
- `materialized_paths: list[Path] = []` — track designer's writes for rejection-time cleanup.

**Phase 1 — Planner.**
- Context: `build_planner_context(workspace_root, round_id, variant_id)`.
- Spawn: `spawn_role(role="planner", ..., validator=validate_planner_json)`.
- `spawn_counts["planner"] = 1 + result.retry_count`.
- If `verdict != "ok"`: write rj (reason_class = `"spawn-failed"` or `"output-parse-fail"`), commit, return.
- Else: `write_role_scratch("planner", parsed)`.

**Phase 2 — Designer.**
- Context: `build_designer_context(...)`.
- Spawn: `spawn_role(role="designer", ..., validator=validate_designer_json)`.
- On non-ok verdict: write rj, commit, return (same shape as planner).
- On ok: write scratch. **Materialize** designer's output to disk:
  - For each entry in `parsed["evidence"]`: write `evidence/ev-<id>.md` with TOML frontmatter. Track path.
  - For each entry in `parsed["claims"]`: write `variants/nodes/<variant>/claims/cl-<id>.json`. Track path.
  - Apply `parsed["patch_diff"]` to `variants/nodes/<variant>/doc/`. (For v0, the patch_diff is unified-diff text; we use `subprocess.run(["git", "apply", "--cached=false", ...], input=patch_diff)` against the workspace. If apply fails, treat as cross-field failure.) Track touched paths.
- If materialization fails: write rj (reason_class=`"cross-field-fail"`), commit, return. Discard any partially-materialized files.

**Phase 3 — Verifier A (cite enforcement).**
- `r1 = verify_citation_completeness(variants_nodes_root)`.
- `r2 = verify_cite_resolution(variants_nodes_root, evidence_root)`.
- If either has failures:
  - reason_class = `"uncited-claim"` if any completeness failure, else `"dangling-evidence"`.
  - rj body lists each failure's variant, section_path, detail.
  - Action = `"phase-a-fail"`.
  - Discard materialized files (unlink + git checkout HEAD -- for modified). Commit, return.

**Phase 4 — Verifier B (excerpt match).**
- `r3 = verify_excerpt_match(variants_nodes_root, evidence_root, threshold=0.92)`.
- If failures: reason_class = `"cross-field-fail"`. rj body lists each failure's `excerpt_diff`. Action = `"phase-b-fail"`. Discard. Commit, return.

**Phase 5 — Reviewer.**
- Context: `build_reviewer_context(...)`.
- Spawn: `spawn_role(role="reviewer", ..., validator=validate_reviewer_json)`.
- On non-ok verdict: write rj, commit, return.
- On ok: write scratch. If reviewer's `decision == "reject"`: build rj from reviewer's `rejection` field (`reason_class`, `evidence_against`, `supersedable_by`). Action = `"reviewer-rejected"`. Discard materialized files. Commit, return.

**Phase 5.5 — Flow A gating (decision_proposals).**
- Extract designer's proposed_decision payloads from its claims (the ones whose `proposed_decision.id` is not yet in `decisions.json`).
- If non-empty, reviewer's parsed output MUST have a `decision_proposals[]` array with one verdict per proposal.
- Run `apply_reviewer_decision_proposals(proposed, verdicts)`.
- If `status == "any-rejected"`: rj body lists rejected ones. Action=`"reviewer-rejected"`, reason_class=`"proposal-rejected"`. Discard. Commit, return.
- Else (all approved): registry mutation deferred to Phase 7a.

**Phase 6 — Verifier C.**
- Context: `build_verifier_c_context(...)`.
- Spawn: `spawn_role(role="verifier_c", ..., validator=validate_verifier_c_json)`.
- On non-ok: write rj, commit, return.
- On ok: write scratch. If `parsed["verdict"] == "dispute"` OR any `per_claim[].verdict == "dispute"`: rj body lists disputed claim_ids + rationales. Action=`"phase-c-dispute"`, reason_class=`"cross-field-fail"` (v0 reuses this; v0.1 may add a narrower `"verifier-c-dispute"`). Discard. Commit, return.

**Phase 7 — Registry mutations (in rails order).**

7a. **Flow A — register-decision** if Phase 5.5 collected approved proposals:
- `register_decision(goal_toml_path, approved, decisions_json_path=derived_decisions_json_path)`.
- `commit_register_decision(new_decision_ids=[p["id"] for p in approved])`.

7b. **Flow C — apply_canonicalization** for each high-confidence position canonicalization in reviewer's attacks:
- Filter `reviewer.attacks` where `at_type == "propose_canonicalization"` AND `kind == "position"` AND `confidence == "high"`.
- For each, verify the `to` slug is canonical in `registry.data[scope]`. If not canonical, log to actions.jsonl with `reject_reason=invalid-canonicalization-target` and skip.
- For each surviving proposal, call `apply_canonicalization(variants_nodes_root, registry, scope, from_slug, to_slug)`. Collect rewrite records.
- After all canonicalizations: persist updated `canonical_slug_registry.json`, call `commit_canonicalize(rewrites)`.
- Medium/low confidence canonicalizations + Flow B cuts + Flow D decision_id renames → out of scope for v0 (deferred to pending_goal_changes.md sub-project).

**Phase 8 — Final merge commit.**
- `commit_merge(round_id, variant_id, section_paths, claim_paths, attack_paths, evidence_paths)`.
- The commit-msg hook validates scope (≤ patch_max_sections), decided-section immutability, file-set whitelist.

**Phase 9 — Detectors (post-commit, read-only).**
- `collisions = detect_position_collisions(variants_nodes_root)`.
- `asymmetry = detect_decisional_asymmetry(variants_nodes_root, decisions)`.
- `stale = detect_stale_proposals(decisions, introduced_round_map, current_round, threshold)`.
- Results don't trigger commits in this sub-project (morning_brief.md consumption is sub-project 5). Append a `detectors_run` entry to actions.jsonl with the counts.

**Return:** `RoundOutcome(verdict="merge", round_id, variant_id, elapsed_seconds, spawn_counts)`.

### 3.2 Materialization-rollback semantics

When a phase rejects after the designer has materialized files:

- For NEW files (`evidence/ev-*.md`, `claims/cl-*.json`, `attacks/at-*.json`, new sections under `variants/nodes/<variant>/doc/`): `Path.unlink(missing_ok=True)`.
- For MODIFIED files (existing sections): `subprocess.check_call(["git", "checkout", "HEAD", "--", str(path)], cwd=workspace_root)`.
- This is narrower than `git checkout -- .` which could nuke unrelated working-tree changes.

The orchestrator tracks `materialized_paths` per round as a list of `(path, is_new)` tuples.

### 3.3 `run_loop` semantics

**Round-id allocation:**
- Discover the highest existing `rounds/round-*/` directory (scan via `iterdir`, extract the numeric suffix, take max).
- `next_n = max + 1` if any rounds exist, else 1.
- Round IDs are zero-padded: `f"round-{n:06d}"`.

**Variant rotation:**
- `variant_id = f"v-{((n - 1) % variant_count) + 1:03d}"`.
- For variant_count=2: round 1→v-001, 2→v-002, 3→v-001, 4→v-002, ...

**Stop conditions:**
- `len(outcomes) >= max_rounds` (if max_rounds set).
- `time.monotonic() - loop_start >= max_wall_clock_hours * 3600` (if set).
- At least one of the two must be specified; both can be specified (first-to-fire wins).

**Resume behavior:**
- Naturally idempotent through the round-id discovery: if `harness run` is killed mid-round, the partially-written `rounds/<id>/scratch/` directory remains (gitignored, no commit). The next `harness run` starts at the next round-id beyond the highest existing directory. The interrupted round's scratch is leftover artifacts; sub-project 6's `harness resume` does the formal cleanup.

### 3.4 `actions.jsonl` schema

Each line is a JSON object with at least:
- `ts`: ISO-8601 UTC timestamp.
- `event`: one of `"round_start"`, `"spawn_complete"`, `"verifier_complete"`, `"materialize"`, `"rejection"`, `"commit"`, `"detectors_run"`, `"round_end"`.

Per-event additional fields:
- `round_start`: round_id, variant_id.
- `spawn_complete`: round_id, role, verdict, retry_count, elapsed_seconds.
- `verifier_complete`: round_id, verifier ("a" | "b"), failure_count, verdict.
- `materialize`: round_id, evidence_count, claim_count, attack_count, section_count.
- `rejection`: round_id, rj_id, reason_class, failed_phase.
- `commit`: round_id, action, sha (the new commit's SHA, captured via `git rev-parse HEAD` after commit).
- `detectors_run`: round_id, collision_count, asymmetry_count, stale_count.
- `round_end`: round_id, verdict, elapsed_seconds.

### 3.5 Validators (passed to spawn_role)

Module-level functions inside `harness/orchestrator.py`:

- `validate_planner_json(d)` — asserts required fields: round, variant, stance, intent, target_sections. Raises ValueError on shape mismatch.
- `validate_designer_json(d)` — asserts patch_diff (str), evidence (list of dicts), claims (list of dicts). Each evidence dict has id/confidence/citations/claim/excerpt; each claim dict roundtrips through `Claim.from_dict`.
- `validate_reviewer_json(d)` — asserts decision in {accept, reject}, rationale, optional rejection dict, optional attacks (each roundtripping through `Attack.from_dict`), optional decision_proposals (each roundtripping through `DecisionProposalVerdict.from_dict`).
- `validate_verifier_c_json(d)` — asserts round, variant, verdict in {confirm, dispute}, per_claim list of dicts with claim_id + verdict + rationale.

These re-use the existing dataclass `from_dict` validators where possible — no duplication.

## 5. Edge cases

- **Empty workspace (no prior rounds):** `run_loop` starts at round-000001.
- **Mid-round crash before any commit:** the leftover `rounds/<id>/scratch/` dir is gitignored; no state corruption. Next `harness run` invocation starts at the next round-id, leaving the orphaned scratch (sub-project 6 cleans it up).
- **Variant has no claims dir yet (first round on that variant):** `_walk_sections` returns empty; verifiers pass trivially; round proceeds.
- **Reviewer returns decision=accept but rejection field present:** structural violation; treated as `output-parse-fail`.
- **Designer's patch_diff is empty:** materialization is a no-op; round proceeds with no doc changes; merge commit stages only claims/attacks/evidence. The commit-msg hook's scope check passes (0 doc sections ≤ N).
- **All four roles passed but no actual changes to commit:** `git commit` with no staged changes errors. The orchestrator handles this by not calling commit_merge if `git diff --cached --quiet` returns 0 (no staged changes). Returns verdict="merge" anyway (the round was successful, just empty).
- **`max_rounds=0`:** run_loop returns empty list without invoking any round.
- **`max_wall_clock_hours=0`:** same as above (or stops after the first round if it's already running).
- **Tool not on PATH (e.g., codex absent):** spawn_role returns verdict=spawn-failed; the round records spawn-failed and continues to the next round (which uses the next variant — same tool failure if config didn't change, so the run will likely fail rapidly through max_rounds). The CLI prints a hint on the first spawn-failed if the configured tool's binary doesn't resolve.

## 8. Success criteria

This sub-project ships when all of these are true:

1. `from harness.orchestrator import run_round, run_loop, RoundOutcome` and `from harness.round_ledger import write_rejection, append_actions_log, ...` work.
2. `python3 -m harness run --rounds 1` against a freshly-`harness init`-scaffolded workspace executes one full round (mocked spawn_role for tests; real claude/codex for the live overnight run that's deferred to operator).
3. A successful round produces 1-3 commits with correct Action trailers (register-decision → canonicalize → merge, in that rails order, with the latter two optional).
4. A round rejected at any phase produces exactly 1 commit (the rejection commit, failure-class Action) with the appropriate rj-*.md.
5. `actions.jsonl` contains an entry per significant event in append-only line-delimited JSON.
6. `run_loop` correctly rotates variants and stops at the dual caps.
7. All ~294 tests pass under `python3 -m unittest discover tests/`.
