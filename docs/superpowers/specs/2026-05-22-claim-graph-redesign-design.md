## 0. Why this redesign exists

The approved v0 design uses a `polarity: supports | refutes | neutral` field on each `cl-*.json` plus a mechanical collision detector keyed on shared `evidence_id`. Pressure-testing during brainstorming surfaced two structural problems:

1. **Polarity has no coherent referent.** The constitution says "Mark polarity for the section's central claim," but each `cl-*.json` IS the section's central claim (per atomicity rule). Polarity-to-self is circular. Both natural readings ("polarity to section question" and "polarity to cited evidence") fail to make collision detection do real epistemic work: the first only fires when sections are framed as yes/no questions, the second mostly catches fabrication.
2. **Shared-evidence_id is the wrong collision coordinate.** The most interesting epistemic disagreements are exactly where variants reach for *different* evidence to make incompatible claims. A detector keyed on shared evidence misses this case entirely.

A third, related gap: the original design has no signal for **cross-variant coverage asymmetry** — one variant addressing a topic the other ignores. The polarity mechanism never could distinguish "silence because not-yet-written" from "silence because affirmatively-out-of-scope," and the latter is a strong epistemic signal worth surfacing.

This redesign replaces polarity with a **decision registry** (`[[decision]]` table in `goal.toml`) and decouples collision detection from evidence overlap. Variants' claims are compared by `decision_id` + `position`, regardless of which evidence they cite. Out-of-scope becomes an affirmative claim type. Slug drift is corrected autonomously overnight via reviewer-proposed canonicalization with append-only safety invariants.

## 1. Problem framing

The harness's research bet (P3 in the parent design doc) is that the variant DAG with stance rotation does *ensemble verification* — variants attacking each other's claims catch what mechanical Verifier A/B miss. This bet only pays out if there's a working mechanism for the harness to *detect* productive disagreement.

The mechanism must:

- **Detect cross-variant disagreement on the same decision**, even when variants cite different evidence
- **Surface affirmative out-of-scope as a signal** distinct from "not yet written"
- **Operate autonomously overnight** without accumulating a human-approval backlog
- **Survive slug drift** (variants naming the same decision/position with slightly different slugs)
- **Stay auditable** — every mutation is a git commit reversible by `git revert`

The polarity + shared-evidence_id design satisfied none of these well. This redesign satisfies all five.

## 2. Design overview

```
goal.toml [[decision]]                            ←  human-curated registry of decisions the doc must address
        ↓ (read at round start)
derived/decisions.json                            ←  orchestrator-built registry view (status: open | proposed | retired)
        ↓ (loaded into CONTEXT.md for each role)
designer authors cl-*.json {decision_id, position, claim_type, [proposed_decision]}
        ↓
reviewer authors at-*.json with at_type ∈ {dispute_claim, propose_decision_cut, propose_canonicalization}
        ↓
Verifier C confirms per-claim verdict (unchanged from parent spec for v0)
        ↓ (after all gates pass)
orchestrator applies auto-applicable mutations:
   - new decisions reviewer approved → goal.toml append + goal_version bump
   - high-confidence position canonicalizations → cl-*.json rewrite + canonical_slug_registry update
        ↓
collision + asymmetry + stale-proposal detectors run on derived/
        ↓
morning_brief.md re-rendered
        ↓
human ritual (between rounds): reviews pending_goal_changes.md for decision cuts,
   medium/low position canonicalizations, and all decision_id canonicalizations
```

The split between **autonomous** and **human-gated** mutations is the central design call: additive operations (register decision, canonicalize position toward existing canonical) auto-apply; destructive or cascading operations (cut a decision, rename a decision_id) wait for the human.

## 3. Schema changes

### 3.1 `goal.toml` — decision registry

Add `[[decision]]` array of tables. Each entry:

```toml
[[decision]]
id = "retry-policy"
question = "How should transient failures be retried?"
status = "open"               # open | proposed | retired
introduced_at = "g-01"        # goal_version when introduced
```

- `status: open` — fair game for designer claims
- `status: proposed` — designer flagged via `proposed_decision`, reviewer approved, orchestrator auto-registered; treated as `open` for designer purposes
- `status: retired` — human cut between rounds; new cl-*.json may not reference; existing decided sections referencing it get retagged `unresolved`

Goal version (`g-NN`) bumps whenever the decision list changes (additions, status flips).

### 3.2 `cl-*.json` — claim schema

- **DROP**: `polarity`, `contradicts`
- **ADD** (required): `decision_id` — kebab-case slug; must match a registered `open` or `proposed` decision OR be paired with `proposed_decision`
- **ADD** (required when `claim_type != out_of_scope`): `position` — kebab-case slug capturing this variant's answer (e.g., `"expo-backoff"`)
- **ADD** (required when `claim_type == out_of_scope`): `out_of_scope_rationale` — text
- **ADD** (optional): `proposed_decision: {id, question, rationale}` — present iff `decision_id` is not yet registered
- **MODIFY** `claim_type` enum: `decision | observation | inference | out_of_scope | unresolved`
  - `out_of_scope` — affirmative declaration that this decision should not be in the doc
  - `unresolved` — designer reached the section but doesn't yet have a defensible position (replaces the "vacuous slug" temptation; see §6 Edge Case C)

Slug regex: `^[a-z][a-z0-9-]*[a-z0-9]$` (kebab-case ASCII, min 2 chars).

Cross-field validation (in `harness.py` + duplicated in pre-commit hook per the existing inline-duplication-over-import-coupling decision):
- Every `decision_id` resolves to a registered (open/proposed) decision OR `proposed_decision` is present
- Every `evidence_ids` reference resolves to an `ev-*` in this round's designer output or a prior `evidence/` file (unchanged)
- Every `[^ev-*]` cite in `patch_diff` body has matching evidence entry (unchanged)
- `claim_type: out_of_scope` requires a registered `decision_id` (you can't propose-and-out-of-scope in one round)
- `position` slug not in the vacuous-slug blocklist (see §6 Edge Case C)
- `position` slug, if not already canonical for this decision, must not be in the alias map (see §5)

### 3.3 `at-*.json` — attack schema

- **ADD** (required): `at_type` enum: `dispute_claim | propose_decision_cut | propose_canonicalization`
- **For `dispute_claim`**: keeps existing fields (`target_claim_id`, `target_variant`, `argument`, `evidence_ids`)
- **For `propose_decision_cut`**: `target_decision_id` (must be registered + not retired), `rationale`
- **For `propose_canonicalization`**:
  - `kind: "decision_id" | "position"`
  - `scope: "<decision_id>"` (required iff `kind == "position"`)
  - `from: "<old slug>"`
  - `to: "<canonical slug>"`
  - `confidence: "high" | "medium" | "low"`
  - `rationale: "<text>"`

### 3.4 `reviewer.json` — top-level shape addition

Reviewer's existing decision (`accept | reject`) and `rejection`/`attacks` arrays unchanged. New field:

- `decision_proposals: [{proposed_id, verdict: "approve" | "reject", rationale}]` — one entry per `proposed_decision` in this round's designer output. Required exactly when the designer round had `proposed_decision` claims.

Rejection of any proposal fails the round with `reason: proposal-rejected`.

## 4. Orchestrator workflow

### 4.1 Flow A — Designer proposes a new decision (autonomous)

1. Designer's `cl-*.json` includes `decision_id` not currently in `derived/decisions.json` AND a `proposed_decision: {id, question, rationale}` payload.
2. Schema validator passes (proposed_decision satisfies the "registered OR proposed" rule).
3. Round proceeds through Verifier A → Verifier B → Reviewer → Verifier C.
4. If reviewer's top-level `decision` is `reject` (constitutional reasons, evidence problems, etc.) → round fails as `reviewer-rejected`, unchanged from parent spec; the rest of this flow does not run.
5. Reviewer's `decision_proposals` array gates each designer-proposed new decision individually. If ANY proposal is `reject` → round fails with `reason: proposal-rejected`, rejection `rj-*.md` written, patch dropped.
6. If all proposals are `approve` AND Verifier C passes (confirm verdict) → orchestrator commits a single `Action: register-decision` commit listing all newly-registered decision_ids in the trailer:
   - Appends each approved decision to `goal.toml` with `status: open`
   - Bumps `goal_version` (e.g., `g-03` → `g-04`)
   - Regenerates `derived/decisions.json`
7. Patch and cl-*.json commit as a subsequent `Action: merge` commit. Per-round commit sequence for a successful round with new-decision proposals and high-confidence position canonicalizations is therefore: `register-decision` → `canonicalize` (if any from §4.3) → `merge`. Each is its own commit with its own trailer. The parent spec's "one commit per logical state transition" invariant is preserved (each Action is one logical transition).

### 4.2 Flow B — Reviewer proposes a decision cut (human-gated)

1. Reviewer writes `at-*.json` with `at_type: propose_decision_cut`.
2. Orchestrator validates `target_decision_id` is registered + not retired.
3. Appends to `pending_goal_changes.md` under "Cuts proposed" header (cut payload + reviewer rationale).
4. Round commits normally; cut proposal does not block.
5. **Between rounds (human ritual)**: human reads `pending_goal_changes.md`, decides; if accepting the cut, edits `goal.toml` (flips status to `retired`), bumps `goal_version`.
6. On next round start, orchestrator detects goal_version bump, regenerates `derived/decisions.json`, runs **section retag walker**:
   - Walks `variants/*/doc/`
   - For each section whose central claim's `decision_id` is now retired, flips section tag from `decided` to `unresolved`
   - Commits as `Action: registry-sync`

### 4.3 Flow C — Position canonicalization (autonomous, gated by confidence)

1. Reviewer writes `at-*.json` with `at_type: propose_canonicalization`, `kind: "position"`, `scope: "<decision_id>"`, `from`, `to`, `confidence`.
2. After all verification gates pass, orchestrator processes the proposal:
   - Looks up `to` in `derived/canonical_slug_registry.json[scope]["canonical"]`
   - If `to` is canonical AND `confidence: high` → **AUTO-APPLY** (see §4.4 below)
   - If `to` is canonical AND `confidence: medium | low` → queue in `pending_goal_changes.md` under "Position canonicalizations (medium/low)" sub-section for human review
   - If `to` is NOT canonical (in aliases, or absent) → REJECT proposal as invalid; log to `actions.jsonl` with `reject_reason: invalid-canonicalization-target`; round still commits

### 4.4 Auto-apply canonicalization mechanics

- Walks all `cl-*.json` files across all variants; rewrites `position: from → to` where `decision_id == scope`
- Updates `derived/canonical_slug_registry.json`:
   - Removes `from` from `canonical[scope]`
   - Adds `aliases[scope][from] = to`
- Commits as `Action: canonicalize` with trailer listing all rewrites + the registry update

### 4.5 Flow D — Decision_id canonicalization (human-gated)

Identical to Flow B in shape: reviewer's `at-*.json` (kind: `decision_id`) appended to `pending_goal_changes.md`. Human edits goal.toml to register the rename, bumps `goal_version`, orchestrator runs registry-sync + rewrites cl-*.json/at-*.json fields. v0 keeps this human-gated because decision_id rewrites cascade across the entire registry.

### 4.6 Between-rounds human ritual (the morning workload)

Single file: `pending_goal_changes.md`, organized by sub-section. Accumulates between rounds:

- **Cuts proposed** (Flow B output)
- **Position canonicalizations (medium/low)** (Flow C medium/low branch)
- **Decision_id canonicalizations** (Flow D output)

New decision registrations (Flow A) are NOT in this list — they auto-applied overnight. The human reviews additive activity post-hoc via the morning_brief but does not need to take action.

## 5. Detectors and append-only canonical slug registry

### 5.1 `derived/canonical_slug_registry.json` (the structural invariant)

```json
{
  "retry-policy": {
    "canonical": ["expo-backoff", "linear-no-backoff"],
    "aliases": {"exponential-backoff": "expo-backoff"}
  },
  "auth-strategy": {
    "canonical": ["oauth2-pkce"],
    "aliases": {}
  }
}
```

Invariants (enforced by orchestrator and pre-commit hook):

1. The `canonical` list is **append-only**. Entries are added when a designer authors a brand-new position for a decision; never removed by edit, only migrated to `aliases` (as alias-key) on canonicalization.
2. The `aliases` map is **append-only as keys**. Once `aliases["exponential-backoff"] = "expo-backoff"` is registered, the key is never re-pointed.
3. Canonicalization is **structurally one-way**: a `propose_canonicalization` is valid iff `to` is currently in `canonical[scope]`. Novel `to` slugs are rejected as invalid (designer adds new canonicals naturally just by authoring them).
4. **Incoming alias rewriting**: when a `cl-*.json` is being committed with `position` = a current alias key, orchestrator rewrites to canonical before writing. Pre-commit hook also enforces (any unwritten alias = reject).

**Oscillation is unrepresentable.** No runtime "lookback N rounds" needed; the invariant is the safety property.

### 5.2 Three mechanical detectors (run after each round's commits)

**Detector 1 — Position collisions**: group cl-*.json by `decision_id` across variants; if `position` slugs differ string-equality → write `collisions/co-*.json` with the decision_id, per-variant `{cl_id, position, evidence_ids}`, `confirmed: false`.

**Detector 2 — Decisional asymmetry**: for each registered decision, compute per-variant status from latest cl-*.json: `unaddressed | out_of_scope | decided`. If two variants disagree (specifically the `decided` vs `out_of_scope` case) → entry in `derived/coverage_asymmetry.json`. Unaddressed silence does NOT trigger.

**Detector 3 — Stale proposals**: walk `derived/decisions.json`; any `status: proposed` for >5 rounds (configurable) → flag in `coverage_asymmetry.json` under `stale_proposals`.

All three are pure Python, no LLM calls.

### 5.3 `morning_brief.md` sections (revised)

- **Survived adversarial review** (unchanged from parent spec)
- **Still weak** (unchanged)
- **Position collisions** (top-level): rows = decisions where variants chose different positions; columns = per-variant position + cl_id + evidence list
- **Decisional asymmetry** (top-level): rows = decisions where one variant `decided` and another `out_of_scope`; columns = per-variant status + (position or rationale). Top-level placement because affirmative scope disagreement is the strongest cross-variant epistemic signal.
- **Canonicalizations applied this round** (new): list of every `Action: canonicalize` commit in the run, with from/to/scope and the cl-*.json files touched. Always shown when non-empty — human needs to be able to audit overnight rewrites quickly on resume.
- **Pending registry changes** (new): three sub-tables — cuts proposed, medium/low position canonicalizations, decision_id canonicalizations
- **Stale proposals** (new, only when non-empty)
- **Rejected this run** (unchanged)
- **Score trajectory** (unchanged)
- **What I'd ask you to look at first** (unchanged, but heuristic now ranks: contested decisions > decisional asymmetry > regressed scores > stale proposals)

## 6. Constitution + CONTEXT.md updates

### 6.1 Constitution.md (replaces the parent spec's polarity-shaped bullets)

```markdown
## Judgment rules for all roles (registry-aware)

- Authoring cl-*.json:
  - For every `decided`-tagged section: produce one cl-*.json with a `decision_id`
    from the registered decisions list AND a `position` slug capturing this
    variant's answer.
  - If the section resolves a question not in the registry: set decision_id to
    a new kebab-case slug AND include `proposed_decision: {id, question, rationale}`.
    Reviewer will gate.
  - If the question is out of scope: set `claim_type: out_of_scope` and provide
    `out_of_scope_rationale`. Silence is uninformative; explicit out_of_scope
    is the strongest cross-variant scope signal.
  - Read derived/decisions.json before authoring. Match existing decision_ids
    and positions when you mean the same concept; invent new slugs only when
    you genuinely differ.

## Slug discipline (decision_ids and positions)

- Slugs are kebab-case ASCII: ^[a-z][a-z0-9-]*[a-z0-9]$. Hook rejects otherwise.
- Position slugs MUST be substantive — they describe the variant's actual answer.
  "tbd" or "unclear" is a designer failure; use claim_type: unresolved if you
  genuinely don't have a position yet.
- If a slug already appears in derived/decisions.json or in another variant's
  claims under the same decision_id, MATCH IT when you mean the same concept.
  Don't invent variations.
- Slug drift across variants is a designer failure, not a stylistic preference.
  Reviewers will flag it as propose_canonicalization. Repeated drift signals
  weak registry hygiene in the designer prompt or in CONTEXT.md construction.

## Reviewer posture

- Default toward rejection when evidence is thin. (unchanged)
- 30-70% accept rate target. (unchanged)
- Write at-*.json `at_type: dispute_claim` when you disagree with another
  variant's claim that cites overlapping evidence. State the alternative inference.
- Spot slug drift across variants. If you see "expo-backoff" in one variant and
  "exponential-backoff" in another under the same decision_id, write at-*.json
  `at_type: propose_canonicalization`. Mark `confidence: high` ONLY when:
    (a) you are certain both slugs mean the same concept, AND
    (b) the canonical slug ("to") is already in the canonical list elsewhere
        in the registry for this decision.
  Otherwise medium or low. High-confidence proposals auto-apply overnight;
  medium/low queue for the morning human. Conservative > aggressive.
- Spot off-thesis decisions. If a registered decision shouldn't be in the doc
  (out of scope, redundant, dead-end), write at-*.json `at_type:
  propose_decision_cut`. Always human-gated; queue for review.
- Gate designer-proposed new decisions strictly. Default deny EXCEPT during
  bootstrap (registry_size < 5), where default permissive: approve unless
  the proposal is clearly off-thesis or duplicates an existing registered decision.
  Approved proposals auto-register overnight — say yes only when you mean it.

## Verifier C posture (unchanged for v0)

- Read the doc patch and cited evidence files only.
- Output per-claim verdict (confirm | weak | dispute) + one-line rationale.
- v0: do NOT attempt to confirm collisions or canonicalizations. That's v0.1.
```

### 6.2 CONTEXT.md template additions (per role)

- **Planner**: + "Registered decisions" list (id, question, status); + "Stale proposals" warning if any.
- **Designer**: + "Registered decisions" list; + "Positions you have already committed to in your variant" (per-decision); + "Pending proposals you previously authored"; + slug discipline reminder.
- **Reviewer** (heaviest grow): + "Registered decisions"; + **"All positions in use across all variants for each decision"** (critical for drift detection); + "Pending designer proposals from this round" (so the gate verdict isn't surprise); + "Recent Action: canonicalize commits from the last 5 rounds" (so reviewer doesn't propose oscillating renames); + `registry_size: N` parameter (used for the bootstrap carve-out).
- **Verifier C**: + "Registered decisions" for context; no new responsibility.

Reviewer CONTEXT.md is the heaviest grow. v0 makes no preemptive mitigation; if the all-positions table grows unwieldy at long horizon, v0.1 will revisit (options at that time: summarize positions older than N rounds via a periodic Claude pass, or hard-cap the table at top-K decisions by recency).

## 7. Edge cases

### A — Verifier C verdict orders before registry mutations

Updated round state machine ordering: registry mutations (auto-register decisions, auto-apply canonicalizations) happen ONLY after all verification gates pass. New step ordering for a successful round, with each labeled commit a separate git commit:

```
... Verifier A → Verifier B → Reviewer → Verifier C
   ↓ all pass
[commit: Action: register-decision]   (if any reviewer-approved proposals this round)
   ↓
[commit: Action: canonicalize]        (if any high-confidence position canonicalizations this round)
   ↓
apply patch → compute scorecard
   ↓
[commit: Action: merge]               (always, the round-completion commit)
```

A disputed claim never leaves an orphaned registry entry because register-decision is gated behind Verifier C. The split into separate commits preserves the parent spec's "one commit per logical state transition" invariant — each Action is one logical transition with its own closed-vocab trailer; auditors can `git log --grep "^Action: register-decision"` to see exactly when the registry grew.

### B — Canonicalization oscillation: solved structurally

See §5.1. Append-only `canonical_slug_registry.json` makes oscillation unrepresentable. No runtime lookback or time-windowed ban needed.

### C — Vacuous position slugs

Two-layer defense:

1. Pre-commit hook has a blocklist: `tbd, unclear, unknown, not-decided, not-yet-decided, na, none, n-a` (plus underscore variants). Cl-*.json with a blocklisted position slug → round fails with `reason: vacuous-position`.
2. Constitution adds the substantive-slug requirement; new `claim_type: unresolved` enum value gives designers a legitimate escape hatch.

### D — Bootstrap permissiveness for empty registry

Orchestrator passes `registry_size` in reviewer's CONTEXT.md. Constitution adds bootstrap carve-out (see §6.1 reviewer posture): default permissive on new-decision proposals when `registry_size < 5`; default-deny once the registry has critical mass. Threshold configurable in `harness.toml`.

### Other handled implicitly

- **Dueling proposals across rounds**: handled by CONTEXT.md regeneration; round N+1's designer sees round N's just-registered decision and uses it without re-proposing.
- **Auto-canonicalization merges two distinct positions**: morning_brief surfaces all `Action: canonicalize` commits; human runs `git revert <sha>` if wrong. The canonical_slug_registry invariant ensures revert restores both data and invariant in one step.
- **Goal_version race condition mid-round**: parent spec already handles via round manifest capturing goal_version at round start.
- **Section retag walker hits missing cl-*.json**: walker is defensive (logs warning, skips section); does not abort.

## 8. Scope & LOC budget

The redesign expands v0 runtime Python past the original "<1000 LOC" leanness target. Approximate net additions:

| Component | LOC |
|---|---|
| Schema changes (dataclasses, validators, duplicated pre-commit) | ~80 |
| Flow A auto-registration logic + section retag walker | ~70 |
| Flow C/D canonicalization (autonomous + human-gated) + at_type handling | ~80 |
| Append-only canonical_slug_registry + alias rewriting + pre-commit enforcement | ~50 |
| Three detectors (collision, asymmetry, stale proposals) | ~70 |
| Morning_brief render updates | ~30 |
| Constitution + CONTEXT.md template population code | ~30 |
| Edge case D bootstrap registry_size templating | ~10 |
| Edge case C vacuous-slug blocklist | ~10 |
| Tests for the above | ~250 |

Estimated total: **~430 LOC of runtime + ~250 LOC of tests** added to the original v0 budget. New v0 total: roughly 1200-1400 LOC of runtime Python (was 700-1000).

**This bump is accepted.** The original 1000-LOC number was a vibe target ("read in 20 min"). 1400 LOC of focused, well-bounded code remains readable. The mechanism quality earns the lines. The "lean version" of the parent spec was caught in a bloating cycle during /office-hours; the user's explicit reframe pulled it back. This redesign earns its way back partway because each piece carries weight: the registry IS the artifact, the detectors ARE the signal, the canonical slug registry IS the safety invariant.

If LOC pressure becomes acute during implementation, the natural split point is `claim_graph.py` — a separate module owning all of §3-§5. The original spec already permits a `verifier.py` split if `verifier.py` grows past ~200 LOC; the same precedent applies. Each file stays under the 20-min-read bar even if total is higher.

## 9. Migration from current v0 design

This redesign supersedes the parent spec's claim graph design. Concrete differences:

| Aspect | Parent spec | This redesign |
|---|---|---|
| Polarity | `supports \| refutes \| neutral` per claim | DROPPED |
| Contradicts | Array of cl-ids on each claim | DROPPED |
| Collision detection coordinate | Shared `evidence_id` | `decision_id` + `position` mismatch |
| Out-of-scope signal | None | `claim_type: out_of_scope` with rationale |
| Decision_id concept | Implicit (= section_id) | Explicit, registered in goal.toml |
| Slug drift | Not addressed | Autonomous canonicalization + append-only registry |
| Canonicalization confirmation | N/A | v0 auto-applies high-confidence positions; v0.1 adds Verifier C confirmation |
| Decision cuts | N/A | Reviewer-proposed; human-gated |
| `at_type` field | N/A (attacks were single-purpose) | `dispute_claim \| propose_decision_cut \| propose_canonicalization` |
| New `claim_type` values | N/A | `out_of_scope`, `unresolved` |
| New action values | N/A | `register-decision`, `canonicalize`, `registry-sync` |
| New goal.toml structure | `[goal]` only | + `[[decision]]` array of tables |
| New derived/ files | INDEX.md, scorecard_rollup.md | + `decisions.json`, `canonical_slug_registry.json`, `coverage_asymmetry.json`, `pending_goal_changes.md` |

The week-1 implementation work (dataclasses + hooks + workspace_template + harness init) from the parent spec is unchanged in shape but the dataclass set grows by ~5 dataclasses to cover the new schemas. The week-2 work (one round, full pipeline) is also unchanged in shape; the validation work grows to include cross-field checks for the new fields. The week-3 work (second variant + Verifier C + claim graph) grows the most — it now includes the canonicalization machinery and the three detectors. Week 3-4 (morning brief + crash recovery + first overnight) absorbs the morning_brief schema expansion.

Rebalanced sequencing recommendation: keep the 3-4 week shape from the parent spec, but stretch week 3 into a full 2-week block (week 3 = registry + claim graph + canonicalization; week 4 = morning brief + crash recovery + first overnight). New estimate: **4-5 weeks for v0**.

## 10. What's NOT in v0 (deferred to v0.1+)

- **Verifier C confirms canonicalization decisions**: v0 trusts the reviewer; v0.1 adds a Verifier C confirmation pass for position canonicalizations (cross-model check that the merger is semantically valid).
- **Verifier C confirms position collisions**: parent spec already deferred; unchanged here.
- **Decision_id canonicalization auto-apply**: v0 always queues; v0.1 may add a high-confidence auto-apply with stricter rails (e.g., `to:` must be canonical, no cl-*.json under `from:` for >N rounds, etc.).
- **Slug fuzzy-matching as a periodic Claude pass**: if the reviewer-integrated canonicalization (Flow C) misses too much drift in real overnight, v0.1 adds a dedicated canonicalizer role spawned every K rounds.
- **Reviewer CONTEXT.md size mitigation**: if the all-positions-across-variants table grows unwieldy at long horizons (50+ rounds), v0.1 adds either a periodic summarization pass or a top-K cap.
- **Goal-declared decision_ids enforced as closed namespace**: v0 lets the registry grow autonomously via Flow A; v0.1 may add an `allow_proposals: false` mode in goal.toml for projects where the decision list is fully known up front.

## 11. Open questions remaining

1. **Initial canonical slug ranking when two variants author distinct slugs in nearby rounds**: v-001 in round N writes `position: "expo-backoff"`; v-002 in round N+1 (CONTEXT.md just regenerated) sees `expo-backoff` in the registry but writes `position: "exponential-backoff"` anyway because designer prefers it. Both slugs become canonical in round N+1's commit. Reviewer in round N+2 proposes canonicalizing one toward the other. Which slug should the reviewer prefer? v0 answer: reviewer prefers the slug that was canonical earlier (chronologically first). Not encoded in the prompt; left to reviewer judgment with the "recent canonicalizations" context. v0.1 may add an explicit "earliest-canonical-wins" heuristic if reviewer judgment proves inconsistent.
2. **Per-decision evidence ledger pruning**: as a decision accumulates many cl-*.json files across many rounds, the morning_brief's per-decision tables grow. v0 shows the latest cl-*.json per variant per decision; older ones are accessible only via `git log`. Acceptable for v0 readability but may need a "show last K positions per decision" mode in v0.1.
3. **What if the human edits goal.toml but doesn't bump goal_version**: parent spec assumes goal_version bump triggers registry-sync. If the human forgets, the orchestrator's view of the registry could diverge from goal.toml. v0 mitigation: at every round start, the orchestrator parses goal.toml, regenerates the in-memory `decisions.json` representation, and compares it byte-for-byte against the on-disk `derived/decisions.json`. If they differ: if `goal_version` also differs, run registry-sync as planned; if `goal_version` is unchanged but content differs, fail loudly with "goal.toml was modified without a goal_version bump; refusing to proceed. Edit goal.toml to bump goal_version, then resume." This avoids mtime-based detection (unreliable across git operations) and forces explicit versioning.

## 12. Success criteria delta

The parent spec's Success Criteria 1-5, 7, 8 are unchanged. Criterion 6 is restated for this redesign:

**SC6' (Variants disagree productively):** At least one of the following appears in a real overnight run, not synthetic test data:
- (a) A `collisions/co-*.json` produced by Detector 1 (same decision_id, different positions across variants), OR
- (b) A `derived/coverage_asymmetry.json` entry where one variant is `decided` and another is `out_of_scope` for the same decision_id, OR
- (c) A `propose_canonicalization` at-*.json that surfaces real slug drift between variants

At least one must trace to a decision the variants genuinely treated differently — not an accidental slug mismatch that the canonicalizer should have caught.

Additionally:

**SC9 (new — Autonomous overnight integrity):** A real 8-hour overnight run with at least 50 variant-rounds completes with zero human-required-between-rounds events on the canonicalization path. That is: every position canonicalization that fires is either auto-applied (high confidence, `to` canonical) or queued for morning review (medium/low confidence) — never blocks the round loop. `pending_goal_changes.md` may accumulate entries; the human handles them only on resume, not during the run.
